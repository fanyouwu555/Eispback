### 1.2 发现的问题

**P1-1 [低] 用户行为日志表 (`usr_user_behavior_log`) 仅有实体和 Mapper，无 Service/Controller**
- 文档 3.6.1 要求记录用户在前端的关键行为（登录登出、创建项目、发起模型调用、充值等）。
- 当前表结构已存在，但没有任何代码写入该表。
- **建议**: 在 `AuthController` 登录成功/登出时、用户创建项目时、调用模型时写入行为日志。

**P1-2 [低] 时长调整未校验原因长度**
- 文档 3.1.1 要求"所有时长调整操作必须填写调整原因（5-200字符）"。
- 当前 `AdjustDurationRequest` 的 `reason` 字段没有长度校验。
- **建议**: 添加 `@Size(min = 5, max = 200)` 校验。

## 二、3.3 通知消息管理模块

### 2.1 测试覆盖

| 测试项 | 结果 | 说明 |
|--------|------|------|
| 创建消息（草稿） | PASS | 支持标题、内容、消息类型、推送范围、推送方式、过期时间 |
| 消息列表查询 | PASS | 支持按类型、状态、时间范围筛选，置顶消息优先排序 |
| 消息详情 | PASS | 返回完整信息及推送目标用户列表（含用户名和阅读状态） |
| 执行推送 | PASS | 全员推送采用分页分批处理（每批1000条），指定用户/角色推送解析目标用户 |
| 撤回消息 | PASS | 校验"只有已发送才能撤回"，撤回后用户关联记录标记为已撤回(readStatus=3) |
| 置顶/取消置顶 | PASS | `POST /{id}/top?isTop=1/0` |
| 我的消息列表 | PASS | 前端用户视角查询 |
| 未读消息数 | PASS | 返回未读数量 |
| 消息类型枚举 | PASS | 7种类型：系统公告、版本更新、模板更新、任务结果、时长预警、充值通知、违规提醒 |

### 2.2 发现的问题

**P2-1 [高] 归档功能逻辑错误**
- `MsgNotificationServiceImpl.archiveNotification()` 第217行将状态设为 `STATUS_SENT`（已发送）。
- 归档应该把消息标记为归档状态，但当前数据库/代码中没有定义归档状态。
- **影响**: 归档接口实际上只是重新设为"已发送"，功能完全失效。
- **建议**: 在 `MsgNotification` 中增加 `isArchived` 字段，或在状态枚举中增加归档状态。

**P2-2 [中] 缺少短信和 App 推送渠道**
- 文档 3.3.1 要求支持"站内信、短信和 App 推送"，一次消息可选择多个渠道同时推送。
- 当前实现仅支持站内信（写入 `msg_user_notification`）。
- **建议**: 增加渠道字段和对应的渠道发送逻辑（至少预留接口）。

**P2-3 [中] 定时推送未实现延时队列**
- 文档 3.3.1 要求"定时推送通过延时队列实现"。
- 当前实现只是记录 `pushTime`，没有实际的定时执行机制。
- **建议**: 使用 Spring 的 `@Scheduled` 定时扫描或集成 Redis 延迟队列。

**P2-4 [低] 消息列表缺少90天归档过滤**
- 文档 3.3.1 要求"消息铃铛默认展示最近90天内的消息记录，超过90天的消息自动进入归档状态"。
- 当前 `listUserNotifications` 没有按时间过滤。
- **建议**: 在查询用户消息时增加 `createdAt >= NOW() - 90 days` 的过滤条件。

**P2-5 [低] 消息创建未记录 `createdBy`**
- 文档 3.3.2 要求 `notifications.created_by` 字段记录创建人管理员ID。
- 当前 `createNotification` 没有从 SecurityContext 获取当前用户并写入该字段。
- **建议**: 在保存消息前设置 `notification.setCreatedBy(currentAdminId)`。

## 三、3.4 模板管理模块

### 3.1 测试覆盖

| 测试项 | 结果 | 说明 |
|--------|------|------|
| 模板列表查询 | PASS | 分页查询，支持名称/场景/状态筛选 |
| 模板详情 | PASS | 返回基础信息+当前版本+历史版本+文件结构 |
| 修改模板信息 | PASS | 更新名称、场景、描述、排序权重、预览图 |
| 上下线切换 | PASS | `POST /{id}/status?status=1/2` |
| 删除模板 | PASS | 逻辑删除 |
| 公共模板列表 | PASS | 前端接口，仅返回上线状态的模板 |
| 文件结构查询 | PASS | 返回当前版本文件树（相对路径） |
| 版本管理 | PARTIAL | 支持上传新版本、回滚，但缺少文件哈希校验 |

### 3.2 发现的问题

**P3-1 [中] 模板存储使用本地文件系统，未对接对象存储**
- 文档 3.4.1 要求"模板资源文件统一存储在对象存储服务中（如 MinIO、阿里云 OSS），通过 CDN 加速分发"。
- 当前 `TemplateStorageServiceImpl` 基于本地磁盘路径 `template.upload-path` 存储。
- **建议**: 将 `TemplateStorageService` 抽象为接口，提供本地实现和对象存储实现（通过配置切换）。

**P3-2 [中] 缺少模板使用统计接口**
- 文档 3.4.1 要求"用户使用统计从多个维度分析模板的使用情况：基础统计（使用次数、创建项目总数）、热门排行、场景分布"。
- 当前 `TplTemplate` 实体有 `usageCount` 和 `projectCount` 字段，但没有任何接口暴露这些数据。
- **建议**: 增加统计查询接口，如 `GET /api/v1/templates/statistics`。

**P3-3 [低] 模板 Entity 缺少 `createdBy` 字段**
- 文档 3.4.2 要求 `templates.created_by` 字段记录创建人管理员ID。
- 当前实体中没有该字段。
- **建议**: 在 `TplTemplate` 中增加 `createdBy` 字段。

**P3-4 [低] 模板版本缺少 `fileHash` 字段**
- 文档 3.4.1 要求"校验通过后计算文件哈希值防止重复上传"。
- 当前 `TplTemplateVersion` 实体没有 `fileHash` 字段。
- **建议**: 增加字段并在上传时计算 SHA-256 哈希。

**P3-5 [低] 模板预览图片缺少独立管理接口**
- `TplTemplatePreviewImage` 实体和 Mapper 已存在，但没有 Controller 接口。
- 当前图片只能通过 `CreateTemplateRequest.previewImage` 传入一个 URL。
- **建议**: 增加预览图片的上传和管理接口。
